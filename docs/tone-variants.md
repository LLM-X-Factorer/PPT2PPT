# Tone 力度参数设计 · v0

**问题来源**：v0 真实数据验证暴露——同一份 PPT 对内（诚实自评）和对外（投资人/合作伙伴）需要不同力度的标题。"营收压在爆品上,其他三层都未独立跑通"对内是诚实，对外是踩雷。

**核心思路**：一份数据 × 三档力度 = 三份适配不同场景的 PPT。**事实不变,修辞滑动。**

## 三档定义

| 档 | 适用场景 | 修辞特征 |
|---|---|---|
| `internal` | 部门内部、对一号位、复盘会 | 尖锐 / 容许黑话 / 容许自嘲 / 数据可负面 |
| `balanced`（默认） | 跨部门、对上级、合作伙伴 | 中性 / 事实判断 / 不带情绪 / 留余地 |
| `external` | 投资人、客户、媒体、市场发布 | 建设性 / 把问题包装成机会 / 把"还没"说成"持续" |

## 力度旋钮影响什么

**只影响**：title / subtitle / 目录描述
**不影响**：items 内文、KR 内容、甘特任务名、产品名/数据/价格

理由：items 是事实，事实不能为了力度变形。变形的应该是判断的"语气"，不是判断的"内容"。

## 力度的语义维度

旋钮其实是三个维度同步滑动：

1. **判断词尖锐度**：硬伤 / 卡点 / 抓手 · 困局 / 问题 / 挑战 · 没 / 待 / 在
2. **时态选择**：完成时（已是问题） / 进行时（正在发生） / 将来时（即将解决）
3. **归因方向**：自我归因（"我们没做好"）/ 中性事实 / 机会归因（"市场正在变化"）

internal 三个维度都向"尖锐 + 完成时 + 自我归因"靠；external 三个维度都向"建设 + 将来时 + 机会归因"靠。

## v1 工具化时的 prompt 模板

```
你是 PPT 顾问。基于以下页面数据,为同一份事实生成三档力度的 title 与 subtitle。

页面数据:
{page_data}

输出 JSON:
{
  "internal":  {"title": "…", "subtitle": "…"},
  "balanced":  {"title": "…", "subtitle": "…"},
  "external":  {"title": "…", "subtitle": "…"}
}

约束:
1. 三档共享同一份事实判断,差别只在修辞
2. internal 容许行业黑话 / 自嘲 / 数据负面
3. balanced 中性,可发给跨部门
4. external 把问题包装成机会,用建设性词汇
5. 不要为了力度而捏造事实——三档都必须能被 items 数据支撑
```

## v0 已验证 v1 必修的边界

1. **internal 不能滑成"内部黑话"**：v0 真实数据里"办会破壁"——熟悉的人秒懂、不熟悉的人完全摸不着。internal 档的力度上限是"尖锐",不是"用缩写"。
2. **external 不能滑回"信息墙"**：包装成机会容易回到"传播力 · 流量端 · 双轮驱动"这种空话。external 档必须保留 Action Title 的判断结构,只是把负面词换成建设词,不能把判断稀释成名词列表。
3. **balanced 是默认,不是平均**：balanced 不是 internal 和 external 的中点,而是"事实陈述",有自己的语气特征(中性、有数字、不情绪化)。

## 局限

- 数字硬编码进标题（"3 项已交付 / 5 项进行中"）的力度调整：v1 需要支持模板 + 数据动态填充,否则 external 档会被迫保留 internal 的数字。
- 中文短句的力度边界比英文窄——四字成语稍一调整就跨档。v1 实测时要校准。
- 力度只在标题层,不在 items——这意味着 items 不能太负面,否则力度旋钮失效（标题再 external,items 还是"试用用户走丢在第 3 步"）。**这是 B 修复留下的副作用**：B 改 items 时倾向尖锐,A 时要审一遍 items 是否需要回到中性。
